# 17. Changelog automation

- Status: Accepted
- Date: 2026-06-21

## Context

The repository tracked a hand-maintained `CHANGELOG.md` that drifted out of date,
while the release workflow built its notes from a raw `git log` dump. Commits
already follow Conventional Commits and pull requests are squash-merged with
conventional subjects, so the history carries enough structure to derive both
artefacts automatically.

## Decision

The changelog is generated by git-cliff from the Conventional Commit history,
configured in `cliff.toml` to a Keep-a-Changelog layout that maps `feat` to
Added, `fix` to Fixed, `perf` and `refactor` to Changed, and `docs` to
Documentation, while skipping housekeeping types. git-cliff is pinned in the
locked development dependencies and invoked through `uv run`, so local and
continuous-integration runs use the same version. On a release tag the workflow
regenerates `CHANGELOG.md`, derives the release body from the same commits, and
attaches the changelog to the GitHub release. Release Drafter continues to
maintain the rolling draft release notes from merged pull requests, a separate
view of the same history.

## Consequences

- The changelog reflects the commit history without manual editing.
- Release notes and the changelog file are derived from one source, so they
  cannot disagree.
- Changelog quality now depends on commit-message discipline, which commitlint
  already enforces.
- git-cliff is reproducible from the lock file rather than a floating action.
